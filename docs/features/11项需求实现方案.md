## 1. 撤销：Ctrl+Z / Ctrl+Y、多次撤回

### 产品与 UI 视角

- **现状**：笔记编辑页有「撤回」按钮，调用 `handleRevertToLastSaved`，基于 `undoStack` 做单步撤回；快捷键未绑定 Ctrl+Z / Ctrl+Y。
- **目标**：与主流编辑器一致，支持 Ctrl+Z 撤销、Ctrl+Y 重做，且支持多次操作。
- **体验**：操作可逆、反馈明确（可配合 Toast 提示「已撤销」/「已重做」）。

### 实现方案

#### 现状

- `NoteManager` 已有 `undoStack: { title, content }[]`，每次编辑在 `handleContentChange` 中按 800ms 防抖入栈，最多 50 步。
- `handleRevertToLastSaved` 只做「撤销一步」：`next.pop()` 恢复到上一个快照。
- 无「重做栈」，撤销后再次编辑会清空原可重做内容（符合多数产品行为）。

#### 建议实现

1. **快捷键**（`NoteManager.tsx` 中的 `handleKeyDown`）  
   - Ctrl+Z / Cmd+Z：撤销一步，执行 `handleRevertToLastSaved`（或等价逻辑）。  
   - Ctrl+Y / Cmd+Y：重做一步（需先实现重做栈）。  
   - Ctrl+Shift+Z：与 Ctrl+Y 等价（可选）。

2. **重做能力**（可选）  
   - 新增 `redoStack`，在每次撤销时把「当前状态」推入 `redoStack`。  
   - 重做时从 `redoStack` 取出状态恢复，并将当前状态推入 `undoStack`。  
   - 有新的编辑时清空 `redoStack`。

3. **与现有撤回按钮的关系**  
   - 撤回按钮继续调用 `handleRevertToLastSaved`，与 Ctrl+Z 共用一套 undo 逻辑。

### 技术要点

- `handleKeyDown` 需在 `view === "editor"` 时监听，且在 `document.activeElement` 不是输入框/文本域时，避免覆盖浏览器默认行为（可选：在 editor 内阻止默认，避免与页面其它输入冲突）。  
- 思维笔记已有 `useUndoRedo`，普通笔记可参考其设计，或保持现有「单步撤销 + 重做栈」的简单实现。

### 相关文件

- `src/components/NoteManager.tsx`（`handleKeyDown`、`handleRevertToLastSaved`、`undoStack`）
- ✅ **已实现**：新增 `redoStack` 与快捷键（Ctrl/Cmd + Z/Y），并避免在 Dialog 内拦截浏览器/输入框默认撤销。

## 2. 粘贴：表格、图片、富文本优化

### 产品与 UI 视角

- **现状**：`SegmentedEditor` 内 `onPaste` 已有 HTML→Markdown 的简单转换（加粗、斜体、标题、段落等），但表格、图片未做专门处理。
- **目标**：从 Excel/网页粘贴表格时，尽量转为 Markdown 表格；粘贴图片时支持上传并插入；富文本粘贴结果更干净、可读。

### 实现方案

#### 2.1 表格粘贴

- 解析 `text/html` 中的 `<table>`，使用 `table-utils` 中 `htmlTableToMarkdown` 或类似逻辑转为 Markdown。
- 若 `text/plain` 中含制表符分隔内容，可尝试识别为表格并转换。
- 将结果插入到当前光标位置；若 SegmentedEditor 支持表格段，可考虑插入表格段而非纯文本。

#### 2.2 图片粘贴

- 在 `onPaste` 中检测 `e.clipboardData.files`，筛选 `type.startsWith('image/')`。
- 调用已有的 `onImageUpload` 或等价上传逻辑，获取 URL 后插入 `![](url)`。
- 支持多图批量粘贴（逐个上传、逐个插入）。

#### 2.3 富文本优化

- 扩展 HTML 解析：`<table>`、`<ul>/<ol>`、`<blockquote>` 等转为对应 Markdown。
- 清理无关标签、内联样式，避免冗余 `span` 等。

### 技术要点

- 表格解析可与 `src/lib/table-utils.ts` 的 `htmlTableToMarkdown` 复用或扩展。  
- 图片上传需异步，可先插入占位符，上传完成后替换 URL。

### 相关文件

- `src/components/SegmentedEditor.tsx`（`onPaste`）  
- `src/lib/table-utils.ts`  
- `NoteManager` 中的 `onImageUpload` 或图片上传逻辑
- ✅ **已实现**：表格（HTML `<table>`/TSV）与富文本清理；**图片粘贴未实现**（按约束）。

## 3. 移动端：光标跳动、聚焦丢失排查

### 产品与 UI 视角

- **现状**：移动端软键盘弹起、编辑时存在光标跳动、聚焦丢失等问题，影响输入体验。
- **目标**：输入时光标稳定、焦点不意外丢失；键盘收起后能正确恢复编辑态。

### 实现方案

#### 排查方向

1. **SegmentedEditor 内部**  
   - `requestAnimationFrame` / `setTimeout` 中恢复光标时，是否与 React 渲染时序冲突。  
   - `autoResizeTextarea`、`updateTextSegment` 触发的重渲染是否导致失焦。

2. **滚动与视口**  
   - `editorScrollContainerRef` 在键盘弹起时的滚动恢复逻辑。  
   - `savedScrollTopRef` 的保存与恢复时机是否合理。

3. **visualViewport 与写作模式**  
   - `isMobileWritingMode` 的切换是否触发不必要的重挂载或 ref 变化。

4. **输入法**  
   - 中文输入法组合态（composition）期间，是否因 `onChange` 或 `setSelectionRange` 打断输入。

#### 建议措施

- 在 `onCompositionStart` / `onCompositionEnd` 期间，暂停或简化光标恢复、防抖保存等逻辑。  
- 减少不必要的 `setState` 和 re-render，避免在输入过程中触发 SegmentedEditor 整体重渲染。  
- 用 `scrollIntoView` 或 `scrollTop` 确保当前输入框在可视区域内，但避免过于频繁地滚动。

### 相关文件

- `src/components/SegmentedEditor.tsx`  
- `src/components/NoteEditor.tsx`（`editorScrollContainerRef`、`savedScrollTopRef`）  
- `src/components/NoteManager.tsx`（`isMobileWritingMode` 相关逻辑）
- ✅ **已实现**：在 `SegmentedEditor` 处理中文输入法组合态（composition）期间不做光标强制恢复，降低跳动与失焦。

- **目标**：勾选、保存、删除等关键操作均有适度触觉反馈，增强确认感。

### 实现方案

1. **统一封装**  
   - 创建 `src/lib/haptics.ts`，提供 `vibrateShort()`、`vibrateSuccess()` 等，内部调用 `navigator.vibrate` 并做兼容判断。

3. **兼容**  
   - `navigator.vibrate` 在部分浏览器不可用，需 `if (navigator.vibrate)` 判断，忽略报错。

## 7. 表格：列宽拖拽、首列冻结

### 产品与 UI 视角

- **现状**：SegmentedEditor 与 MarkdownRenderer 中的表格为固定布局，列宽不可调，无冻结列。  
- **目标**：支持拖拽表头边界调整列宽；移动端可固定首列，横向滚动时首列保持可见。

### 实现方案

#### 7.1 列宽拖拽

- 表头每列右侧增加可拖拽的「手柄」，使用 `mousedown` / `touchstart` 监听拖拽。  
- 拖拽时更新列宽（`colgroup` + `col` 的 `width`，或 CSS 变量）。  
- 需考虑：  
  - SegmentedEditor 中表格为受控组件，列宽需作为 state 或存储到 `segment` 的扩展字段。  
  - Markdown 本身无列宽语义，可把列宽存到笔记的 metadata 或自定义数据结构，与表格段绑定。

#### 7.2 首列冻结

- 首列 `position: sticky; left: 0`，并设置 `z-index` 和背景色，避免滚动时透出下层内容。  
- 表格容器需 `overflow-x: auto`，冻结列才能生效。  
- 移动端首列冻结可设为可选（如通过设置开关）。

#### 技术难点

- 列宽需持久化，否则刷新后恢复默认。  
- 与现有分段表格数据结构（`Segment`）的扩展需谨慎设计，避免破坏兼容性。

### 相关文件

- `src/components/SegmentedEditor.tsx`（表格渲染与编辑）  
- `src/components/MarkdownRenderer.tsx`（预览态表格）  
- `src/lib/table-utils.ts`（若需扩展表格数据结构）

## 8. 最近打开（类似 Notion 最近访问列表）

- **现状**：无最近访问记录。  
- **目标**：在 Dashboard 或笔记列表顶部展示「最近打开」的笔记，支持点击快速进入。

### 实现方案

#### 8.1 数据层

- **存储**：本地 `localStorage` 或 IndexedDB，key 如 `sumunote:recent-notes:{userId}`。  
- **结构**：`{ noteId, title, folderId, lastOpenedAt }[]`，按 `lastOpenedAt` 降序，最多保留 20 条。  
- **更新**：进入编辑页或打开笔记详情时，将该笔记加入/更新最近列表。

#### 8.2 UI

- 在 Dashboard 或 NoteList 顶部增加「最近打开」区块，展示最近 5–10 条。  
- 每条显示标题、文件夹路径（可选）、最后打开时间。  
- 点击跳转到对应笔记编辑页。

#### 8.3 与离线

- 离线时仍可展示本地已有的最近记录；若笔记已删除，打开时提示并可从列表中移除。

## 9. 长文：虚拟滚动 / 懒加载

### 产品与 UI 视角

- **现状**：整篇内容一次性渲染，超长笔记（如数万行）可能导致卡顿。  
- **目标**：长文场景下保持流畅滚动和编辑体验。

### 实现方案

#### 9.1 虚拟滚动（阅读态）

- 使用 `@tanstack/react-virtual` 或 `react-window` 对 Markdown 渲染结果按「块」虚拟化。  
- 需将 Markdown 解析为块列表（段落、标题、表格、列表等），按块渲染，只渲染可视区域及缓冲区。  
- 难点：Markdown 解析与块拆分，需与现有 `MarkdownRenderer` 结合。

#### 9.2 SegmentedEditor 懒加载

- 若段落/表格段数量很多，可对非可视区域的段做懒渲染（如仅渲染前 N 段 + 视口附近）。  
- 滚动时动态挂载/卸载段，需维护段索引与滚动位置的映射。

#### 9.3 策略

- 先设定阈值（如 content 长度 > 50KB 或段数 > 100）再启用虚拟化，避免对短文引入额外复杂度。  
- 编辑态虚拟化难度更高，可先实现阅读态（预览）的虚拟滚动。

### 相关文件

- `src/components/MarkdownRenderer.tsx`  
- `src/components/SegmentedEditor.tsx`  
- 新增 `VirtualizedMarkdown` 或类似组件

## 10. 离线首屏：优先展示缓存内容

### 产品与 UI 视角

- **现状**：有 `offline-storage`、`pending_sync`，但首屏仍以网络请求为主。  
- **目标**：离线或弱网时，优先从本地缓存渲染笔记列表和内容，再在后台尝试同步。

### 实现方案

#### 10.1 缓存策略

- **笔记列表**：在 IndexedDB 或 localStorage 中缓存当前用户的笔记列表（id、title、updated_at 等摘要）。  
- **笔记内容**：对最近打开或当前文件夹下的笔记做本地缓存。  
- **加载顺序**：先读缓存渲染首屏，再发起网络请求；请求返回后增量更新 UI。

#### 10.2 实现要点

- 扩展 `offline-storage` 或新建 `cache-storage`，支持读写笔记列表与单篇内容。  
- `NoteManager` 中 `fetchNotes`、`fetchSubFolders` 先读缓存，再 `fetch`；若有缓存，可立即 `setNotes` 等，无需等待网络。  
- 需处理缓存过期、与云端冲突时的策略（如「本地优先」或「网络优先」可配置）。

### 相关文件

- `src/lib/offline-storage.ts`  
- `src/lib/note-service.ts` 或 `NoteManager` 中的数据加载逻辑

## 11. 启动速度：路由预加载、关键资源预取

### 产品与 UI 视角

- **现状**：Next.js 默认按需加载，首次进入 Dashboard、笔记页可能有延迟。  
- **目标**：减少首屏等待，提前加载可能访问的页面与资源。

### 实现方案

#### 11.1 路由预加载

- 使用 `next/link` 的 `prefetch`（默认在视口内会预取）。  
- 在 Dashboard 对「常用入口」如 `/dashboard`、`/notes/[id]` 使用 `<Link prefetch>`。  
- 可对最近打开的笔记在后台执行 `router.prefetch('/notes/[id]')`。

#### 11.2 资源预取

- 对关键 JS chunk 使用 `<link rel="preload">` 或 Next.js 的 `next/script` 策略。  
- 字体、关键 CSS 尽早加载，避免 FOUT/FOIT。

#### 11.3 数据预取

- 在 Dashboard 加载时，可预请求当前用户笔记列表的接口，减少进入笔记列表后的等待。  
- 结合「最近打开」，预加载最近笔记的内容。

### 相关文件

- `src/app/dashboard/page.tsx`  
- `src/app/layout.tsx`  
- `next.config.js`（若有自定义配置）
