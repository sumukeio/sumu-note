### 先帮你翻译一下你遇到的现象

**你现在的体验大概是这样：**

- **刚进入编辑器时**：你点到要写的地方，屏幕上能看到光标。
- **一开始输入几个字之后**：屏幕突然「跳」了一下，你眼前只剩下笔记的最上面几行内容，**光标和你正在输入的那一行完全看不到了**。
- 你继续打字，其实文本是在往下某个地方被输入，只是你每次看到的都是「笔记开头」，好像界面老是在帮你「滚回顶部」。

从用户的角度，就是：**一旦进入「真正写作」状态，整个编辑区就会强行回到顶部，把正在编辑的行和光标挤出视野**。

### 这个问题在代码里的根源（结合当前实现）

我简单看了下你现在的实现，核心相关的几块是：

- `NoteEditor` 里有一个 `editorScrollContainerRef`，是**包住整个正文的可滚动容器**，移动端时高度大概是  
  `h-[calc(100dvh-14rem)]` + `overflow-y-auto`。
- `SegmentedEditor` 里面把正文拆成多个 `Textarea`，并做了很多「**移动端编辑后恢复光标/滚动位置**」的逻辑（`savedCursorStateRef` / `restoreCursorPosition` 等），还有一个**每 50ms 轮询的 effect**，会在有待更新内容时触发 `onChange` 和光标恢复。
- 顶层还有一个 `ViewportVars` 组件，通过 `visualViewport` 写 CSS 变量（`--vv-bottom-inset` 等），用来给 `scrollPaddingBottom`，避免键盘遮住底部。

结合这些逻辑，你现在遇到的「只要开始编辑就只看到顶部」很大概率是这几件事叠加导致的：

- **移动端键盘弹出后，浏览器会主动调整可视区域**，但我们的正文又放在一个固定高度的内层滚动容器里（`h-[calc(100dvh-...] + overflow-auto`），这本身就容易产生「浏览器想让光标可见，组件又在强制维持某个高度/滚动位置」的冲突。
- `SegmentedEditor` 在每次输入时：
  - 会保存光标和滚动位置、
  - 改变 `Textarea` 高度（自适应高度）、
  - 通过轮询 + 定时器在某些时机尝试**恢复光标位置和内部滚动**。
- 这些恢复逻辑在桌面端是为了避免光标抖动，但在某些移动端浏览器里，**很容易和系统自己的自动滚动叠加出「跳回顶部」这样的副作用**：  
  系统刚把光标行滚进可视区，我们的代码又根据「上一次保存的滚动状态」做了一次恢复，结果让容器回到了一个不正确的位置（通常就是接近顶部的位置）。

简单说，就是：

> **系统在「努力帮你看到光标」，而我们的滚动恢复逻辑在「努力把东西恢复到之前认为正确的位置」——两边打架，最后的结果就是你看到的「一编辑就只剩顶部」**。

### 有没有「彻底」的解决办法？

可以有，而且是可以在你当前架构下落地的，关键是换一种思路：

- **不再试图「强行恢复整个编辑器的滚动状态」**（尤其是在键盘刚弹出那一段时间）。
- 改为：**只在需要的时候，让当前光标所在的那一行进入可视区域**，并且尽量让浏览器的默认行为来主导。

一个相对稳妥的方案会包含几步（给你一个方向感）：

- **移动端专门路径**：
  - 当检测到移动端（`/iPhone|Android/`）时：
    - 减少/移除 `SegmentedEditor` 里那套「轮询 + 自动恢复光标/滚动」的逻辑，至少在移动端不做这件事。
    - 不再对内部 `Textarea` 强行设置 `scrollTop`，而是让它自然跟随。
- **聚焦/输入时只做「让光标行可见」这件事**：
  - 给 `SegmentedEditor` 增加一个回调，比如 `onCaretRectChange(rect)`，在 `onFocus` / `onChange` 时计算出当前光标在 `editorScrollContainerRef` 中的相对位置。
  - 在 `NoteEditor` 里，用这个 `rect` 做**单次的「scrollIntoView 类似效果」**：如果光标底部快贴近容器底部，就稍微向下滚一点；绝对**不要**每 50ms 一直「修正」。
- **简化高度控制，减少嵌套滚动**：
  - 移动端上尽量保证「**只有一个主要滚动容器**」：就是包着 `SegmentedEditor` 的这层 `div`；避免再在里面制造新的 `overflow`。
  - 继续利用你已有的 `ViewportVars` + `scrollPaddingBottom`（这一点其实是对的），只负责保证底部有足够空白给键盘，而不是频繁恢复滚动。

这样做的结果应该是：

- 键盘弹出时，由浏览器主导把焦点行滚到一个合适位置。
- 我们的代码只在「系统没有完全滚好」时，轻轻往上/下调整一点点，**不会再把整个内容卷回顶部**。
- 因为不再轮询恢复滚动状态，你在「持续打字」的时候，视图会保持在光标附近，而不是不停跳动。
